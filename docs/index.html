Crystal: Open and Extensible Software-Defined Storage for OpenStack Swift
Crystal allows you to deploy new storage functionalities in Swift thanks to an extensible data plane and manage their behavior via a dynamic control plane.
IFTTT-like SDS policies
Crystal enables administrators to write SDS policies in a "If-This-Then-Than" (IFTTT) form via a DSL. These policies can change the behavior of the system by enforcing storage filters ("That"). The enforcement of filters may be static or dynamic ("This"). A user-friendly dashboard enables administrators to write policies and control the system.
Flexible Introspection
The "This" part of Crystal's IFTTT-like policies refers to introspective information. Crystal enables to define dynamic policies based on monitoring information and the metadata of object requests. Moreover, inspection metrics can be plugged-in and exposed to the DSL at runtime.
Transparent Interception
In Crystal's IFTTT-like policies "That" means storage filters. Crystal enables administrators to transparently intercept object requests with storage filters. Besides, developers can implement many types of storage filters, ranging from storage management techniques to performance control.
Understanding Crystal: A Simple Example
Crystal is devised to ease storage management in the cloud via Software-Define Storage models. To understand how easy it is to manage your OpenStack Swift cluster, we show a simple example policy as they are defined in the Crystal dashboard, as well as how it is enforce on object requests.
Architecture & Overview
Title example
Subtitle example
Content example: In Crystal, administrators provision SDS services to tenants via policies. To open the control plane, Crystal includes a metaprogramming DSL which is able to incorporate customizable instrospective information in form of metrics, and very appealingly, that can be extended at runtime without side effects on the system operation. Further, it offers and innovative distributed model for the controller. As a distinguishing feature, distributed controllers can be auto-generated by the DSL itself for simple rule-based IFTTT-like policies. In addition, the control loop is extensible: Developers can easily incorporate new metrics to capture the core aspects of the input workflows that can be consumed by new controllers.
Content example: The data plane executes arbitrary computations and transformations on objects flows to satisfy storage policies or recipes. It provides two core extension points: The inspection triggers and interception filters. With the inspection triggers, Crystal provides distributed controllers with introspective information on the state of the system at runtime. This allows the control plane to dynamically respond to changes in input workloads in real time. Rich data plane programmability is delivered through the interception filters, which intercept object flows in a transparent manner and run computations on them. An external third party integrating a new filter only needs to contribute the logic; the deployment and execution of the filter is managed by Crystal.
